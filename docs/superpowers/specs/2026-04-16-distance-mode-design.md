# Distance Mode — Design Spec (Multi-Camera)

## Overview

Add distance-based zone classification to SafetyVision. Three cameras (back, left, right) mounted on the forklift capture the surroundings. Each camera has its own homography calibration. Detected persons' footpoints are transformed from pixel coordinates to real-world meters in a forklift-relative frame. The **closest person across all cameras** determines the alert level.

Uses `supervision.ViewTransformer` for the homography transform. The existing horizontal band mode is preserved as a fallback for single-camera deployments.

## Architecture: Zone Strategy Pattern

Extract zone classification from `InferenceWorker` into a strategy interface with two implementations:

- **BandZoneStrategy** — current Y-band logic (`yellow_start_y` / `red_start_y`), single-camera only.
- **DistanceZoneStrategy** — multi-camera homography. Holds one `ViewTransformer` per camera, takes detections tagged by `camera_id`, transforms each through the right calibration, picks the minimum distance across all detections.

`InferenceWorker` delegates zone classification to whichever strategy is configured.

### Multi-camera pipeline

```
Camera Back ──┐
Camera Left ──┼──► 3 Capture Workers ──► frame queue ──► 1 Inference Worker (Hailo)
Camera Right ─┘         (Option A)         (tagged         │
                                          by camera_id)     ▼

                                                          │
                                                          ▼

                                              (Distance: aggregates
                                              across all cameras)

                                                          ▼
                                                Decision ──► Alert
```

from dataclasses import dataclass
from typing import Protocol

@dataclass
class ZoneResult:
    zone_level: str                     # "danger" | "medium" | ""
    distance_m: float | None            # None for band mode, meters for distance mode

        self,
        detections_by_camera: dict[str, list],  # {camera_id: [Detection, ...]}
        frame_h: int,
        frame_w: int,
    ) -> ZoneResult: ...

## DistanceZoneStrategy

```python
# src/safetyvision/zones/distance.py
import json
import supervision as sv
import numpy as np

class DistanceZoneStrategy:
    def __init__(
        self,
        calibrations: dict[str, str],  # {camera_id: calibration_path}
        danger_m: float,
        warning_m: float,
        smoothing_frames: int = 3,
    ):
        self.transformers: dict[str, sv.ViewTransformer] = {}
        for camera_id, path in calibrations.items():
            data = json.load(open(path))
            self.transformers[camera_id] = sv.ViewTransformer(
                source=np.array(data["source_points"], dtype=np.float32),
                target=np.array(data["target_points"], dtype=np.float32),
            )
        self.danger_m = danger_m
        self.warning_m = warning_m
        self._distance_buffer: list[float] = []
        self._smoothing = smoothing_frames

    def classify(self, detections_by_camera, frame_h, frame_w) -> ZoneResult:
        min_dist = float("inf")
        min_camera = None

        for camera_id, detections in detections_by_camera.items():
            if not detections or camera_id not in self.transformers:
                continue
            footpoints = np.array(
                [((d.x1 + d.x2) / 2, d.y2) for d in detections], dtype=np.float32
            )
            world_pts = self.transformers[camera_id].transform_points(footpoints)
            distances = np.linalg.norm(world_pts, axis=1)
            cam_min = float(distances.min())
            if cam_min < min_dist:
                min_dist = cam_min
                min_camera = camera_id

        if min_camera is None:
            self._distance_buffer.clear()
            return ZoneResult(zone_level="", distance_m=None, source_camera=None)

        # Temporal smoothing: median over last N frames
        self._distance_buffer.append(min_dist)
        if len(self._distance_buffer) > self._smoothing:
            self._distance_buffer.pop(0)
        smoothed = float(np.median(self._distance_buffer))

        if smoothed <= self.danger_m:
            zone = "danger"
        elif smoothed <= self.warning_m:
            zone = "medium"
        else:
            zone = ""

        return ZoneResult(zone_level=zone, distance_m=smoothed, source_camera=min_camera)
```

### Temporal smoothing

Raw frame-to-frame distance jitters due to detection noise (bbox edges shifting by a few pixels). The strategy keeps a rolling buffer of the last N frames' minimum distance and returns the **median**. Default N=3 — enough to dampen jitter without adding perceptible alert latency (at 15 FPS, 3 frames = 200ms).

When no detections are present, the buffer is cleared so the next person appearing gets an immediate reading, not a stale smoothed value.

### Runtime performance

`ViewTransformer.transform_points()` is a `cv2.perspectiveTransform` matrix multiply. For 3 cameras with typical 0-3 detections each, total cost is <0.05ms per frame. No measurable impact on the pipeline.

## Coordinate Frame (all cameras)

All target coordinates are in a **forklift-relative frame**, shared across all cameras:

- **Origin (0, 0)** = the forklift center (or a fixed reference point on the forklift)
- **X axis** = lateral (positive = right side of forklift's forward direction)
- **Y axis** = longitudinal (positive = forward direction)

**Each camera's calibration maps its own pixel view to this same forklift-centered coordinate frame.** This is critical — it's what lets us compare distances across cameras.

When calibrating, the operator measures each ground point's X/Y relative to the forklift center, not relative to the camera. The calibration UI shows a diagram of the forklift with axes and explicitly labels which direction is X and Y.

Since all three cameras share the origin, a person detected anywhere around the forklift has a consistent "distance from forklift" value regardless of which camera sees them.

## Calibration File Format (per camera)

```json
{
  "camera_id": "back",
  "source_points": [[160, 96], [480, 96], [128, 256], [512, 256]],
  "target_points": [[-2.0, -1.5], [2.0, -1.5], [-2.5, -3.5], [2.5, -3.5]],
  "frame_width": 640,
  "frame_height": 480,
  "created_at": "2026-04-16T12:00:00Z"
}
```

- `source_points`: 4 pixel coordinates from this camera's frame
- `target_points`: 4 corresponding forklift-relative coordinates in meters
- `frame_width` / `frame_height`: used to validate source points stay in bounds if camera resolution changes
- Thresholds live in `safetyvision.yaml` only — not duplicated here

## Calibration Web UI

### Page: `/calibration`

Camera selector at top (Back / Left / Right), then a split layout:
- **Left**: captured frame from selected camera where operator clicks 4 ground points
- **Right**: coordinate input panel — pixel coords auto-filled on click, operator types real-world X/Y meters for each point relative to forklift center. Forklift-axis diagram shown above inputs.
- **Top bar**: Camera selector, "Capture Frame" button, "Save Calibration" button for the selected camera

Each camera is calibrated independently. Once all three are calibrated, a "Enable Distance Mode" button becomes available to switch `zone_mode: distance` and restart the service.

### Calibration frame source

The snapshot for clicking points comes from the **same stream the detector uses** (the 640×480 detection substream). This ensures the pixel coordinates map correctly to runtime frames.

Implementation: the web app requests the latest decoded frame via a shared file on disk. The capture worker writes the most recent frame per camera to `/dev/shm/safetyvision/frame_<camera_id>.jpg` (tmpfs, no disk I/O). The calibration endpoint reads that file.

### API Endpoints

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/api/calibration/frame?camera=<id>` | Latest JPEG snapshot from named camera |
| `GET` | `/api/calibration?camera=<id>` | Current calibration for camera (404 if none) |
| `GET` | `/api/calibration/status` | Returns which cameras have calibration, current zone_mode |
| `POST` | `/api/calibration?camera=<id>` | Validates, saves calibration_<id>.json |
| `POST` | `/api/calibration/enable` | Validates all 3 calibrations exist, sets `zone_mode: distance`, restarts safetyvision |
| `POST` | `/api/calibration/disable` | Sets `zone_mode: bands`, restarts safetyvision |
| `DELETE` | `/api/calibration?camera=<id>` | Removes calibration for one camera |

### Validation on POST /api/calibration

1. Exactly 4 source and 4 target points
2. Source points within `frame_width` × `frame_height`
3. **Non-collinearity check**: compute the 3×3 homography matrix via `cv2.findHomography`. Reject if:
   - `cv2.findHomography` returns `None`
   - The matrix determinant is below `1e-6` (degenerate)
4. **Sanity check**: transform the 4 source points back; mean reprojection error must be < 2 pixels

### Validation on POST /api/calibration/enable

- All 3 cameras (back, left, right) have valid calibration files
- `danger_threshold_m > 0` and `< warning_threshold_m` in config

## Applying Calibration

Web UI (`safetyvision-ui`) and detection pipeline (`safetyvision`) run as separate systemd services. They cannot share memory.

Flow when operator clicks "Enable Distance Mode":

1. Web app verifies all 3 calibration files exist and validate
2. Web app updates `zone_mode: "distance"` in `safetyvision.yaml`
3. Web app runs `systemctl restart safetyvision` (same pattern as existing `POST /api/apply`)
4. On startup, `InferenceWorker` reads config, factory creates `DistanceZoneStrategy` loaded with all 3 transforms

Same pattern for disable — flip config, restart.

### Error handling on startup

If `zone_mode: distance` but a calibration file is missing or corrupted:
- Log an error at startup
- Fail fast (exit non-zero) — systemd restart loop will keep trying, but this forces the operator to notice and fix rather than silently falling back
- Alternative considered: auto-revert to bands mode. Rejected because a forklift safety system going into a degraded mode silently is worse than an obvious failure.

### Validation rules

- `zone_mode` ∈ {`"bands"`, `"distance"`}
- If `zone_mode == "distance"`, `calibration_<id>.json` must exist for every camera in `input.cameras`
- `danger_threshold_m > 0`, `< warning_threshold_m`
- `distance_smoothing_frames` ∈ [1, 10]

- **Unit tests**:

  - `DistanceZoneStrategy.classify` — known detections per camera → expected min distance and source_camera
  - Temporal smoothing — verify median over buffer, buffer reset on no-detection
  - Collinearity rejection — 4 collinear points must fail validation

- **Manual tests**:
  - Live calibration UI with each of the 3 cameras

  - Verify correct `source_camera` reported when person moves from one camera's view to another
